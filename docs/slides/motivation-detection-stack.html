Firewalls, access control and segmentation try to prevent the breach. But prevention is never perfect — IoT devices are weakly patched and widely exposed. My IDS is the detection layer: it assumes something got through, and catches it.

~/thesis/motivation — detection_pipeline.sh 02 / 03
Detection = monitor → analyze → report

The SOC workflow — and where my demo maps onto it

Monitor
Capture & extract
tcpdump → dpkt extractor reproduces the dataset's 25 packet-window features per flow.
→
Analyze
Classify
MLP scores each flow: Benign vs Attack, and which of 8 attack families.
→
Report
Alert
Live dashboard raises a prioritized verdict per flow for a human analyst.

This is a detect-and-alert IDS, not an inline blocking gate — exactly how a SIEM / SOC consumes a detection engine: high-quality alarms for a human to triage, not automated traffic drops.

Why machine learning belongs here
“Just tell me if something looks weird. I don't know what weird is — you figure it out.”
— IBM Technology, Cybersecurity Architecture: Detection. ML finds attack patterns rules miss.
Rule Signature: matches a known indicator. Brittle, misses variants.
ML Learned: generalizes across an attack family from data.
Mine Supervised classifier of known attack families — honest scope. Unsupervised novelty detection is future work.
