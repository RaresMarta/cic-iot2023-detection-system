Live Monitoring System

The ids/apps/monitor app is a beside-path network IDS: it passively sniffs traffic on a Linux bridge, classifies it with the same MLP used offline, and enforces bans out of band via the host firewall — it is never in the forwarding path, so the protected site is never touched. A FastAPI server on :7870 streams every decision to a React dashboard over SSE. Three interchangeable packet sources (live / replay / simulate) feed one source-agnostic detector.

01

End-to-End Detection Pipeline

Packets enter from one of three sources, get grouped into 10-packet tumbling windows per unordered host pair, and are turned into the 25 CIC-IoT-2023 features by the exact same extractor.py code used on the offline file path — giving live/offline parity. Each window is then classified by two models: a 2-class gate (the ban signal) and an 8-class family label. A capture thread does all packet-rate work and hands completed windows to an async consumer over a bounded, drop-oldest queue.

02

Per-Window Processing Sequence

What happens to a single completed window, from queue pull to published event. The MLP runs inline, batch-1 in the async consumer. Every window publishes a flow event; the first malicious window from a new attacker also emits an alert; only when the policy returns ban does the enforcer fire (off the event loop, in an executor) and emit a ban event.

Failure isolation: _handle() is wrapped so one bad window never kills the detector. A slow dashboard client only drops its own oldest broker events — it can never apply backpressure to detection or to other clients.
03

Ban Policy & Out-of-Band Enforcement

A ban requires all four gates to pass: gate verdict = Attack, calibrated confidence ≥ 0.60, N consecutive malicious windows from the same source (default 2, so one misclassified window can't ban), and the source not in the NEVER_BAN allowlist. Enforcement adds the attacker IP to two nftables sets with a TTL timeout — the host kernel then drops its forwarded packets until the ban auto-expires. The protected site is never modified.

The gate — not the family — drives the ban. The 8-class family label is informational only. Two nftables tables are required because L2-bridged container-to-container traffic (the demo's attacker↔website) skips the IP forward hook unless br_netfilter is loaded; the bridge family sees it regardless. NEVER_BAN defaults to the mock site (172.30.0.10) and the bridge gateway (172.30.0.1).
04

SSE Event Lifecycle

The detector publishes four event types to the broker, which fans them out to every dashboard client over GET /api/stream. This is the frozen contract the React frontend consumes. An attacker moves through these states as malicious windows arrive and then stop.

ℹ
Other endpoints: /api/health (mode + model + enforcer), /api/stats (flow/malicious/dropped counters, by-family histogram, banned count), /api/blocklist (in-memory bans, no nft shell-out), and /api/inject (queue attack bursts — simulate mode only). A recovered event auto-expiring the nft entry happens via the TTL, independent of the lifecycle task.
05

Beside-Path Deployment Topology

All containers sit on one user-defined bridge (idsnet, 172.30.0.0/16, host interface ids-br0). The detector runs host-networked with NET_RAW (raw capture) and NET_ADMIN (promiscuous mode + nftables), so it sees every frame the bridge forwards between the attacker swarm and the protected site — and writes drop rules on the host kernel. It is beside the path, never inline: attacker↔website traffic flows directly through the bridge; the detector only observes and, when it decides to ban, drops the attacker at the firewall.

✓
Out-of-band by design: the detector is a passive sensor plus active responder, not an inline IPS. It never forwards or proxies traffic, so a detector crash or slowdown cannot break connectivity to the protected site. Enforcement is the only side effect, and it acts on the attacker's source IP at the host firewall with an auto-expiring TTL — the protected host (172.30.0.10) is in NEVER_BAN and is never touched.
